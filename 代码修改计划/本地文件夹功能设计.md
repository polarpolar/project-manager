# 本地文件夹功能设计文档

## 一、功能概述

每个项目对应一个本地文件夹，该文件夹与云端同步，支持多人协作维护项目文件。系统通过 File System Access API 访问本地文件系统，实现文件列表展示、文件上传/删除/预览、以及 AI 分析重要文件等功能。

***

## 二、文件夹命名规则

新建项目文件夹时，按以下规则命名：

```
{项目来源}{日期(YYYYMMDD)}-{项目名称}
```

**字段说明：**

| 字段   | 来源                | 说明                |
| ---- | ----------------- | ----------------- |
| 项目来源 | `channel` 字段      | 为空时直接跳过，不留占位符     |
| 日期   | `contractDate` 字段 | 无合同签署时间时，使用项目录入时间 |
| 项目名称 | `name` 字段         | 完整项目名称            |

**示例：**

```
北化20251125-255所项目
佳电运维20260318-红沿河三期
20260318-无来源项目              ← 项目来源为空时
```

**说明：**

- 项目来源和日期之间**无分隔符**，日期和项目名称之间用 `-` 分隔
- 文件夹名称**创建后固定不变**，不随项目阶段、名称或来源的变化而自动修改
- 系统不依赖文件夹名称来关联项目，用户可以在 OneDrive 里自由改名，不影响系统功能
- 文件夹名称仅作为人类可读的标签，方便在 OneDrive 中识别和排序

## 三、系统文件

每个项目文件夹和根目录下各有一个系统隐藏文件，以 `.` 开头，不在文件列表中显示。

### 3.1 项目标记文件：`.pm-project.json`

位置：每个项目文件夹内

内容：

```json
{
  "projectId": "abc123",
  "createdAt": "2025-03-18T10:00:00Z"
}
```

作用：将文件夹与项目 ID 绑定，是项目↔文件夹关联的**唯一真相来源**，用于兜底查找。

### 3.2 根目录索引文件：`.pm-index.json`

位置：OneDrive 根目录下

内容：

```json
{
  "abc123": "C-2503AB12-红沿河三期",
  "def456": "P-2503CD34-佳电运维项目"
}
```

作用：存储 projectId → 文件夹名 的映射，支持跨设备同步，加速启动时的文件夹查找。

***

## 四、三层存储架构

系统维护三层存储，互为备份：

```
第一层：localStorage
  - 存储内容：projectId → 文件夹名 的映射（字符串，可序列化）
  - 特点：本机缓存，读写最快，刷新页面后仍有效，换电脑后失效
  - 用途：本机快速启动

第二层：OneDrive 根目录下的 .pm-index.json
  - 存储内容：同上，JSON 文件
  - 特点：随 OneDrive 同步，多台电脑共享
  - 用途：跨设备恢复映射

第三层：每个项目文件夹内的 .pm-project.json
  - 存储内容：该文件夹对应的 projectId
  - 特点：只要文件夹存在就有效，不依赖任何外部索引
  - 用途：终极兜底，文件夹改名后仍能找到
```

内存中维护运行时映射：

```js
window.projectDirMap = {
  "abc123": <FileSystemDirectoryHandle>,
  "def456": <FileSystemDirectoryHandle>,
}
```

## 五、核心流程

### 5.1 启动 / 配置根目录时

按优先级逐层查找，建立 `projectDirMap`：

```
1. 读取 localStorage 中的映射
      ↓ 某条记录找不到对应文件夹（文件夹被改名或删除）
2. 读取根目录下的 .pm-index.json
      ↓ 仍找不到
3. 扫描根目录下所有子文件夹，读取 .pm-project.json 匹配 projectId
      ↓ 找到后
4. 自动修复：将新文件夹名同步回 localStorage 和 .pm-index.json
```

### 5.2 新建 / 导入项目时

```
1. 检查根目录是否已授权
      ↓ 未授权
   提示用户配置根目录，用户确认后先授权再继续

2. 扫描根目录下所有【未关联】的文件夹，与项目名称做模糊匹配
   匹配优先级：
     a. 文件夹名完全包含项目名称
     b. 项目名称完全包含文件夹名
     c. 两者字符重叠比例超过阈值

3a. 找到候选文件夹
   → 弹出确认框："检测到可能匹配的文件夹「xxx」，是否关联？"
   → [关联] / [新建文件夹] / [跳过]
   → 用户选择关联：写入 .pm-project.json，更新三层存储

3b. 未找到候选文件夹
   → 自动新建文件夹（按命名规则）
   → 写入 .pm-project.json
   → 更新三层存储
```

### 5.3 打开项目文件面板时

```
1. 用 projectDirMap[projectId] 直接获取文件夹句柄
      ↓ 句柄不存在
2. 触发启动时的逐层查找流程
      ↓ 仍找不到
3. 提示用户："项目文件夹不存在，是否重新创建？"
   → [新建] / [手动关联] / [忽略]
```

### 5.4 手动关联（兜底入口）

在项目编辑页提供"关联本地文件夹"按钮：

```
1. 弹出文件夹选择器，用户手动选择对应文件夹
2. 写入 .pm-project.json
3. 更新三层存储（localStorage + .pm-index.json + projectDirMap）
```

### 5.5 写入时（三层同步）

任何创建或更新关联的操作，都同步写入三处：

```
1. 更新 localStorage
2. 更新根目录下的 .pm-index.json
3. 写入/更新项目文件夹内的 .pm-project.json
```

### 5.6 删除项目时

```
1. 删除 localStorage 里的记录
2. 从 .pm-index.json 里移除该条目
3. 从 projectDirMap 里移除
4. 本地文件夹是否删除由用户自行决定，系统不强制
```

***

## 六、边界情况处理

| 情况                       | 处理方式                                               |
| ------------------------ | -------------------------------------------------- |
| 根目录未授权                   | 提示用户配置，功能降级                                        |
| 文件夹被用户改名                 | 启动时扫描兜底，自动修复索引                                     |
| 文件夹被删除                   | 提示用户重新创建或手动关联                                      |
| 两个文件夹有相同 projectId       | 取扫描到的第一个，显示警告 toast                                |
| 文件夹存在但无 .pm-project.json | 手动关联时写入；自动扫描时忽略                                    |
| 换了一台电脑                   | localStorage 失效，降级读 .pm-index.json（OneDrive 同步后可用） |

## 七、文件显示过滤规则

以下文件不在文件列表中显示：

```js
const displayFiles = files.filter(f =>
  !f.name.startsWith('.')  // 过滤所有隐藏文件，包括 .pm-project.json
);
```

## 八、推荐的修改方案

根据现有的模块划分：

&#x20;

- `file-system-module.js` — 文件系统底层操作（句柄、读写、上传、删除、预览）
- `file-analysis.js` — 文件内容分析（AI 分类、合同/协议/报价解析）

按这个划分，新功能应该这样分配：

**放在** **`file-system-module.js`** **里：**

- `getProjectDirName(project)` — 生成文件夹名称（命名规则）
- `initProjectDirMap()` — 启动时三层查找，建立 projectDirMap
- `createProjectDir(project)` — 新建文件夹并写入 `.pm-project.json`
- `linkProjectDir(projectId, dirHandle)` — 手动关联，更新三层存储
- `getProjectDirById(projectId)` — 替换现有 `getProjectDir`
- `matchExistingDir(projectName)` — 模糊匹配已有文件夹
- `syncIndex()` — 同步写入 localStorage 和 `.pm-index.json`

现有代码里 `getProjectDirName` 被 `file-analysis.js` 多次调用，但这个函数本身不在这两个文件里，应该是在project-module.js里定义的。不需要挪，放在 `project-module.js` 里很合理——它是根据项目数据生成名称，属于项目业务逻辑。只需要把它的实现按新命名规则更新就行。
